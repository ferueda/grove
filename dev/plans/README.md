## Deferred Direction Options

These were grounded by the audit but are product/design choices, not defects. They were not turned into implementation plans in this batch.

- Add CLI pool sizing without config files (`GROVE_MAX_TREES` or equivalent). Evidence: SDK supports `maxTrees`; CLI context currently exposes only repo and Grove dir configuration.
- Document `Grove.stats()` and sync package README stubs. Evidence: SDK exposes `stats()`; root README documents aggregate CLI output but the SDK method and package README stubs lag.
- Spike Windows verified process safety. Evidence: non-Linux/macOS process detection returns `unverified` by design; shipping verified Windows safety would be larger and correctness-sensitive.

## Findings Considered And Rejected

- Broad coverage baseline: Grove already has meaningful lease integration buckets and `pnpm check`; targeted contract tests are higher leverage than adding coverage reporting first.
- Full process-scan cache: worth revisiting later, but duplicative list scanning is a smaller safe first step.
- Default `onHookFailure: "fail"`: behavior-breaking for existing integrators; Plan 004 documents and hardens current semantics instead.
- Hook command shell execution as a vulnerability: hooks are trusted programmatic config by design. Documentation or an exec-file hardened mode can be considered separately, but this batch focuses on actual failure semantics.
- Linux `CLK_TCK = 100` owner detection: plausible portability issue, but lower confidence and lower leverage than the selected safety and state-machine findings.
- Removing legacy state migration: real complexity, but high risk for existing pools and not needed before the selected plans.
- TypeScript native preview migration: current toolchain is intentional and locked by `pnpm-lock.yaml`; document/monitor separately if it causes CI instability.
- Dependabot major update policy: useful maintenance, but lower leverage than CLI/error/state safety work.
